  DataElement
   (SDCDataElement) 		0..*dom-2: If the resource is contained in another resource, it SHALL NOT contain nested Resources
dom-1: If the resource is contained in another resource, it SHALL NOT contain any narrative
dom-4: If a resource is contained in another resource, it SHALL NOT have a meta.versionId or a meta.lastUpdated
dom-3: If the resource is contained in another resource, it SHALL be referred to from elsewhere in the resource
Comments:
Often called a clinical template.

  url 1..1uriComments:
Can be a urn:uuid: or a urn:oid:, but real http: addresses are preferred. Multiple instances may share the same url if they have a distinct version.

  identifier 1..*IdentifierComments:
Typically, this is used for identifiers that can go in an HL7 V3 II (instance identifier) data type, e.g., to identify this data element outside of FHIR, where it is not possible to use the logical URI.

  version 0..1stringComments:
There may be different data element instances that have the same identifier but different versions. The version can be appended to the url in a reference to allow a refrence to a particular business version of the data element with the format [url]|[version].

  status 1..1codeBinding: PublicationStatus (required)
Comments:
Allows filtering of data elements that are appropriate for use vs. not.

This is labeled as &ldquo;Is Modifier&rdquo; because applications should not use a retired data element without due consideration.

  experimental 0..1booleanComments:
Allows filtering of data element that are appropriate for use vs. not. This is labeled as &ldquo;Is Modifier&rdquo; because applications should not use an experimental data element in production.

  date 0..1dateTimeComments:
Note that this is not the same as the resource last-modified-date, since the resource may be a secondary representation of the data element. Additional specific dates may be added as extensions or be found by consulting Provenances associated with past versions of the resource.

  publisher 0..1stringComments:
Usually an organization, but may be an individual. The publisher (or steward) of the data element is the organization or individual primarily responsible for the maintenance and upkeep of the data element. This is not necessarily the same individual or organization that developed and initially authored the content. The publisher is the primary point of contact for questions or issues with the data element. This item SHOULD be populated unless the information is available from context.

  name 1..1stringComments:
The name is not expected to be globally unique. The name should be a simple alpha-numeric type name to ensure that it is computable friendly.

  contact 0..*ContactDetailComments:
May be a web site, an email address, a telephone number, etc.

  telecom 0..*ContactPoint
  useContext 0..*UsageContextComments:
When multiple useContexts are specified, there is no expectation whether all or any of the contexts apply.

  copyright 0..1markdownComments:
null The copyright statement does not apply to values for the data element, only to its definition.

  code 0..*CodingBinding: LOINC Codes (example)
Comments:
The concept SHALL be properly aligned with the data element definition and other constraints, as defined in the code system, including relationships, of any code listed here. Where multiple codes exist in a terminology that could correspond to the data element, the most granular code(s) should be selected, so long as they are not more restrictive than the data element itself. The mappings may be used to provide more or less granular or structured equivalences in the code system.

  definition 0..1markdownComments:
If the definition is omitted, the meaning of the data element must be explicit in either the name of the data element, the definitional concept referenced by the data element or some set of extensions that formally define the meaning.

  comment 0..1markdownComments:
If it is possible to capture usage rules using constraints, that mechanism should be used in preference to this element.

  requirements 0..1markdownComments:
This element does not describe the usage of the element (that's done in comments), rather it's for traceability of <em>why</em> the element is either needed or why the constraints exist as they do. This may be used to point to source materials or specifications that drove the structure of this data element.

  type 0..1Elementele-1: All FHIR elements must have a @value or children
eld-4: Aggregation may only be specified if one of the allowed types for the element is a resource
Comments:
The Type of the element can be left blank in a differential constraint, in which case the type is inherited from the resource. Abstract types are not permitted to appear as a type when multiple types are listed. (I.e. Abstract types cannot be part of a choice).

  code 1..1uriComments:
If the element is a reference to another resource, this element contains &ldquo;Reference&rdquo;, and the targetProfile element defines what resources can be referenced. The targetProfile may be a reference to the general definition of a resource (e.g. http://hl7.org/fhir/StructureDefinition/Patient). There would be one pair of type/code for each allowed target resource type.

  example 0..*Elementele-1: All FHIR elements must have a @value or children
Comments:
Examples will most commonly be present for data where it's not implicitly obvious from either the data type or value set what the values might be. (I.e. Example values for dates or quantities would generally be unnecessary.) If the example value is fully populated, the publication tool can generate an instance automatically.

  label 1..1string
  value[x] 1..1base64Binary, boolean, code, date, dateTime, decimal, id, instant, integer, markdown, oid, positiveInt, string, time, unsignedInt, uri, Address, Age, Annotation, Attachment, CodeableConcept, Coding, ContactPoint, Count, Distance, Duration, HumanName, Identifier, Money, Period, Quantity, Range, Ratio, Reference(), SampledData, Signature, Timing, Meta
  minValue[x] 0..1integer, decimal, instant, date, time, dateTime, QuantityComments:
Except for date/date/instant, the type of the minValue[x] SHALL be the same as the specified type of the element. For the date/dateTime/instant values, the type of minValue[x] SHALL be either the same, or a a <a href="datatypes.html#duration">Duration</a> which specifies a relative time limit to the current time. The duration value is positive, and is subtracted from the current clock to determine the minimum allowable value. A minimum value for a Quantity is interpreted as an canonical minimum - e.g. you cannot provide 100mg if the minimum value is 10g.

  maxValue[x] 0..1integer, decimal, instant, date, time, dateTime, QuantityComments:
Except for date/date/instant, the type of the maxValue[x] SHALL be the same as the specified type of the element. For the date/dateTime/instant values, the type of maxValue[x] SHALL be either the same, or a a <a href="datatypes.html#duration">Duration</a> which specifies a relative time limit to the current time. The duration value is positive, and is added to the current clock to determine the maximum allowable value. A maximum value for a Quantity is interpreted as an canonical maximum - e.g. you cannot provide 10g if the maximum value is 50mg.

  maxLength 0..1integerComments:
Receivers are not required to reject instances that exceed the maximum length. The full length could be stored. In some cases, data might be truncated, though truncation should be undertaken with care and an understanding of the consequences of doing so. If not specified, there is no conformance expectation for length support.

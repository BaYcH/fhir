Operations on the RESTful API

The RESTful API defines a set of common operations performed on a repository of typed resources. These operations follow the RESTful paradigm of managing state by Create/Read/Update/Delete operations on a set of identified resources. While this approach solves many use cases, there is some specific functionality that can be met more efficiently using an RPC-like paradigm. This specification describes an operation framework that seamlessly extends the RESTful API.

Operations have the following general properties:

  • Each operation has a name
  • Each operation has a list of 'in' and 'out' parameters
  • The parameters are either resources, data types or search parameters
  • The operations are subject to the same security arrangements as the RESTful API
  • The URIs for the operation end points are based on the existing RESTful API address scheme
  • The operations may make use of the existing repository of resources in their definitions

Note: the definition of these services does not prevent the use of other kinds of services not dependent on or integrated with the RESTful API.

Defining an Operation

Each Operation is defined by:

  • A context for the operation - system, resource type, or resource instance
  • A name for the operation
  • A list of parameters, with their definitions

For each parameter, the following information is needed:

  • Name - the name of the operation. For implementer convenience, the name should be a valid token (see below)
  • Use - In | Out | Both
  • Type - a data type, a search parameter type, a resource type, or "Document". Out parameters must have a type which is a resource
  • Documentation - describes its use

Note that search parameter types are allowed for use where the parameter is used as a search parameter i.e. to link between the context of operations, and a set of existing resources that the operation refers to, where as data types and resources are used to provide information directly to the operation. Executing an Operation

The URL for an operation end-point depends on its context:

  • system: the url is [base]\$[name]
  • resource type: the url is [base]\[type]\$[name]
  • resource instance: the url is [base]\[type]\[id]\$[name]

An operation is invoked by performing an HTTP POST to the operation end-point. The content of the body is a multipart/form-data that contains the parameters for the operation. Where the parameter type is a data type, the type may be represented as either json or xml; the root name of the object is the name of the parameter.

Operation Output

If the operation succeeds, the HTTP Status code is 200 OK. An HTTP status code if 4xx or 5xx indicates an error. User agents should note that servers may issue redirects etc to authenticate the client in response to an operation request.

If the result of the operation is a single resource, then the body of the response will be the resource. Note that the usual arrangement in regard to content type apply. If the result of the operation is more than one resource, then the server returns an atom feed with the resources.

The resources that are returned by the operation may be retained and made available in the resource repository on the operation server. In that case, the server will provide the identity of the resource in the content-location header in the response, or in the atom feed id/self link. When resources that are not persisted are returned in an atom feed, they should be assigned a cid: identifier that will never be re-used.
